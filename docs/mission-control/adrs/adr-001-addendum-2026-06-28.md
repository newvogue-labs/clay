# ADR-001 Addendum (2026-06-28): execution override expiry в WorkspaceStateSnapshot

Status: Accepted
Extends: ADR-001 (Runtime State Model)
Related: ADR-025 (Execution Layer & Real-Money Gate)
Slice: S-EXEC-3c

## Decision

`WorkspaceStateSnapshot` расширяется двумя полями для фронтового countdown по
активному execution-override. Backend — source of truth: фронт не хранит и не
вычисляет срок локально, а получает абсолютный момент истечения и серверное
время для компенсации расхождения часов клиента.

`execution_mode` и `execution_override_state` уже присутствуют в snapshot
(слайсы 3a/3b). S-EXEC-3c добавляет только срок жизни override
(`execution_override_expires_at`) и серверное время (`server_time`). Наличие
активного override определяется по `execution_override_state`; `expires_at`
управляет исключительно отображением countdown.

## Schema

| Поле | Тип | Null | Описание |
|------|-----|------|----------|
| `execution_override_expires_at` | string (ISO 8601, UTC) | да | Абсолютный момент истечения активного execution-override. `null` — override не активен. |
| `server_time` | string (ISO 8601, UTC) | нет | Серверное время на момент сборки snapshot. Клиент: `offset = server_time - client_now`, countdown = `expires_at - (client_now + offset)`. |

## Consequences

- Countdown переживает reload (источник — backend).
- При `expires_at <= now` фронт делает refetch snapshot и следует ответу
  backend, не флипая состояние локально.
- Часовой дрейф клиента компенсируется через `server_time`.
- Аддитивно: существующие потребители snapshot не ломаются, `expires_at` nullable.
