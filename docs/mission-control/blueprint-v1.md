# CLAY Mission Control v1 — Структурированный инженерный план

Дата: 2026-03-30
Статус: planning blueprint
Формат: build-ready engineering plan

## 1. Назначение документа

Этот документ фиксирует инженерный план `CLAY Mission Control v1` как самостоятельной локальной торгово-аналитической панели, независимой от `OpenClaw`.

Документ предназначен как рабочая основа для разработчика или команды, которая будет собирать систему под ключ. Это не просто идея и не только product-описание: здесь зафиксированы режимы работы, состав модулей, роли ИИ, UX-режимы, потоки данных, fallback-логика, история, audit, risk-control и порядок реализации.

## 2. Цель системы

Построить локальную `web-first` панель управления, которая:

- ведет фоновый мониторинг крипторынка;
- переходит в активный торговый режим в часы работы пользователя;
- получает в realtime рыночные и внешние данные;
- использует несколько ИИ-модулей с разными ролями;
- синтезирует ранжированный список торговых сигналов;
- помогает вручную принимать решения по сделкам на `Binance Spot`;
- сохраняет полную историю, объяснения, решения, результаты и обратную связь;
- помогает улучшать стратегии и версии моделей по итогам demo и реальных сессий.

## 3. Базовые принципы v1

- Система полностью самостоятельна и не зависит от `OpenClaw`.
- Панель работает локально на ПК пользователя.
- Панель не исполняет сделки автоматически.
- Пользователь сам совершает сделки в `Binance`.
- Панель может предлагать действия, но критичные изменения всегда требуют явного подтверждения.
- Система должна переживать сбои внешних `API` и облачных моделей через `degraded mode`.
- `Knowledge base` не должна участвовать в hot path каждого сигнала.
- Вся архитектура строится с приоритетом объяснимости, наблюдаемости и контроля.

## 4. Границы v1

### Входит в v1

- `Binance Spot only`
- `Intraday active` как основной режим
- fallback в `Intraday slow`
- `3–5` рабочих пар одновременно
- выбор shortlist пар системой с подтверждением пользователем
- цикл анализа каждые `10–15 минут`
- горизонт сигнала `30–90 минут`
- `OHLCV + volume + simplified order book`
- внешний контекст: `crypto news + community sentiment`
- `Chief Agent`, `Market Scanner`, `News/Sentiment Agent`, `Forecast Model`
- режимы стратегии: `Trend-following`, `Momentum`, `Defensive`
- `AI console` внутри панели
- `strict risk-control`
- `hard preflight`
- `pre-session briefing`
- `backtesting / replay` в базовом виде
- `post-session analysis`
- `full audit trail`
- `model import + version management`
- `demo/test account` как основной режим обкатки

### Не входит в v1

- auto-execution ордеров
- `Binance Futures`
- multi-user и роли доступа
- публичный внешний доступ к панели
- fully autonomous strategy switching
- fully autonomous model routing
- агрессивный plugin marketplace
- full-scale local MLOps platform
- постоянный cloud-first runtime

## 5. Пользовательский режим эксплуатации

- Основное окно активности пользователя: `09:00–22:00 MSK`
- Внутри окна возможны несколько активных торговых фаз с паузами
- Система должна поддерживать режимы:
  - `background monitoring`
  - `pre-session briefing`
  - `active trading session`
  - `session pause`
  - `post-session review`

Активный торговый режим включается по схеме `schedule + manual control`.

## 6. Пользовательский сценарий v1

### До активной сессии

- Фоновые сервисы собирают рыночный контекст.
- Система формирует shortlist кандидатов.
- Система строит `pre-session briefing`.
- Пользователь проходит `hard preflight`.
- Пользователь подтверждает shortlist, стратегию, модель и запуск активной сессии.

### Во время активной сессии

- Панель показывает ранжированный список сигналов.
- Система обновляет анализ каждые `10–15 минут`.
- Пользователь видит:
  - сигналы;
  - confidence;
  - entry zone;
  - expected move;
  - причины сигнала;
  - новости и sentiment;
  - текущий режим стратегии;
  - риск-предупреждения;
  - таймер до следующего обновления.
- Система может предложить:
  - заменить одну из рабочих пар;
  - сменить режим стратегии;
  - перейти в `Defensive`;
  - остановить или ограничить сессию.
- Пользователь вручную исполняет сделки в `Binance`.

### После сессии

- Панель подтягивает результаты через `read-only API`.
- Система проводит session review.
- Пользователь оставляет обратную связь по сигналам.
- Панель сохраняет это в audit/history и использует в дальнейшем анализе.

## 7. Главные экраны панели

### 7.1 Trading Screen

Основной боевой экран.

Должен содержать:

- активные пары;
- ранжированный список сигналов;
- confidence;
- entry zone;
- expected move;
- краткое объяснение;
- график выбранной пары;
- новости и sentiment по выбранной паре;
- активный режим стратегии;
- предупреждения по риску;
- таймер до следующего анализа;
- статус сигнала: `active`, `weakening`, `invalidated`, `expired`.

### 7.2 Control Center

Экран системного контроля.

Должен содержать:

- статус модулей;
- ошибки и задержки;
- использование CPU / RAM / GPU / storage;
- статус `API`;
- запуск / остановку / перезапуск сервисов;
- переключение стратегии;
- переключение моделей и версий;
- переключение shortlist пар;
- состояние fallback / degraded mode.

### 7.3 AI Console

Диалоговый интерфейс с главным агентом.

Позволяет:

- спросить, почему был дан сигнал;
- сравнить пары;
- запросить предложение по смене стратегии;
- попросить объяснить конфликт между моделями;
- запрашивать summary текущей сессии;
- инициировать actions только через явное подтверждение пользователя.

### 7.4 Session Review Screen

Экран разбора сессии.

Должен показывать:

- историю сигналов;
- историю сделок;
- win rate;
- P&L;
- качество сигналов;
- разрез по стратегиям;
- разрез по парам;
- разрез по времени суток;
- обратную связь пользователя;
- рекомендации, что подкрутить.

### 7.5 Knowledge / Research Screen

Экран базы знаний и research-контекста.

Должен содержать:

- личные заметки;
- правила стратегий;
- чеклисты;
- материалы для future expansion;
- позже: книги, PDF, transcripts, curated research.

Важно: этот экран и его данные не должны быть обязательной частью hot path торговых сигналов.

## 8. Логическая архитектура системы

### 8.1 Frontend Layer

- `web-first` UI
- локальная панель в браузере
- позже можно упаковать в desktop shell
- single-user интерфейс

### 8.2 Control / API Layer

Центральный backend, который:

- обслуживает UI;
- агрегирует статус модулей;
- принимает подтверждения пользователя;
- управляет конфигами;
- управляет моделями и версиями;
- выдает историю и audit.

### 8.3 Background Services Layer

Фоновые сервисы, которые живут постоянно:

- `Market Data Service`
- `History Store Service`
- `Session Scheduler`
- `Connector Manager`
- `Audit / Event Store`

### 8.4 On-Demand Analysis Layer

Сервисы, которые работают интенсивнее в активной сессии:

- `Shortlist Engine`
- `Feature Engine`
- `Forecast Inference Service`
- `News/Sentiment Aggregator`
- `Chief Agent Orchestrator`
- `Briefing Generator`
- `Backtesting / Replay Worker`

### 8.5 Knowledge Layer

- `Knowledge Base Store`
- `Research Ingestion`
- `Search / Retrieval`

Использование ограничено:

- для strategy review;
- для post-session analysis;
- для AI briefing;
- для research и настройки;
- не для обязательного вызова в каждом сигнале.

## 9. Роли ИИ в системе

### 9.1 Chief Agent

Главный аналитик и оркестратор.

Отвечает за:

- финальную синтезацию сигнала;
- объяснение сигнала;
- отображение конфликта между модулями;
- снижение confidence при конфликте;
- pre-session briefing;
- рекомендации по смене стратегии.

### 9.2 Market Scanner

Отвечает за:

- обзор рынка;
- shortlist пар;
- оценку рыночной динамики;
- выявление интересных движений.

### 9.3 News/Sentiment Agent

Отвечает за:

- сбор внешнего фона;
- сжатие новостей;
- sentiment summary по активам;
- передачу внешнего контекста в сигнал.

### 9.4 Forecast Model

Отвечает за:

- прогноз направления;
- прогноз силы движения;
- числовой вклад в оценку сигнала.

## 10. Принцип назначения моделей

- Модели назначаются на роли через панель.
- Роли не закреплены жестко в коде.
- Важные смены моделей требуют подтверждения пользователя.
- Новая модель не включается без review-карточки:
  - версия;
  - дата обучения;
  - результаты `backtest`;
  - ключевые отличия;
  - основные риски.

## 11. Логика shortlist пар

Shortlist строится по критериям:

- ликвидность;
- объём;
- волатильность.

Во время активной сессии система может предложить замену пары, но только с подтверждением пользователя.

## 12. Логика сигналов

Каждый сигнал должен содержать:

- `BUY / SELL / HOLD`
- confidence
- entry zone
- expected move
- timeframe
- причины сигнала
- активную стратегию
- риск-статус
- признак конфликта между модулями

Сигналы показываются не потоком без структуры, а ранжированным списком.

## 13. Жизненный цикл сигнала

Сигнал имеет динамический жизненный цикл.

Сигнал инвалидируется или слабеет, если:

- цена ушла слишком далеко от entry zone;
- confidence заметно упал;
- сменился режим рынка;
- пришел более сильный противоположный сигнал;
- достигнут максимальный TTL.

Сигнал не должен “висеть” в интерфейсе без контекстной актуальности.

## 14. Риск-контроль

В `v1` используется строгий уровень риска.

Система должна:

- оценивать риск сигнала;
- выявлять плохое качество данных;
- выявлять конфликт моделей;
- помечать перегретый рынок;
- советовать не входить;
- советовать остановить сессию;
- переводить систему в `Defensive`, если сработали жесткие триггеры.

При жестких risk-триггерах:

- система автоматически уходит в `Defensive`;
- confidence понижается;
- возврат в обычный режим возможен только после подтверждения пользователя.

## 15. Degraded Mode и локальный fallback

Если облачные ИИ-модели или часть внешних `API` недоступны:

- система переходит в `degraded mode`;
- панель обязана явно показать это состояние;
- confidence понижается;
- часть сигналов может блокироваться;
- подключается минимальный локальный fallback.

Локальный fallback включает:

- упрощенную локальную прогнозную модель;
- базовую сигнальную логику;
- отсутствие претензии на полный интеллект облачного режима.

## 16. Demo / Test режим

Основной режим обкатки v1:

- `Binance demo/test account`
- пользователь вручную совершает сделки;
- панель через `read-only API` читает результаты;
- panel review строится по реальным demo-сделкам.

Критерии допуска к осторожной реальной торговле:

- минимум `5` отдельных demo-сессий;
- среди них несколько длинных сессий около `7 часов`;
- стабильный плюс по серии;
- отсутствие крупных просадок;
- отсутствие критических технических сбоев.

## 17. История, audit и обратная связь

В `v1` обязателен полный `audit trail`.

Хранится:

- сигналы;
- объяснения;
- предложения агентов;
- решения пользователя;
- сделки;
- результаты;
- preflight-результаты;
- смены стратегии;
- смены моделей;
- degraded/fallback события;
- команды из `AI console`;
- причины отклонения сигналов;
- обратная связь пользователя.

Пользователь должен иметь возможность оставлять рабочую обратную связь:

- вошла / не вошла;
- почему отклонила;
- был ли сигнал полезным;
- доверяла ли объяснению.

## 18. Локальная история и хранение данных

Система обязана копить собственную локальную историю.

Нужно хранить:

- рынок;
- новости;
- sentiment summary;
- shortlist history;
- сигналы;
- сделки;
- историю моделей;
- историю стратегий;
- историю сессий;
- review и feedback.

## 19. Knowledge Base

В архитектуре `v1` knowledge base присутствует сразу.

Первая рабочая итерация включает:

- личные заметки;
- правила стратегий;
- чеклисты;
- наблюдения пользователя.

Дальнейшее расширение:

- PDF;
- книги;
- transcripts;
- curated research.

Ограничение:

- knowledge base не должна становиться обязательным bottleneck для каждого сигнала.

## 20. Технический runtime-подход

Рекомендуемая модель runtime:

- базовые сервисы живут фоном;
- тяжелые аналитические процессы запускаются по режиму;
- панель является пультом управления всей системой.

Это снижает нагрузку на ПК и делает систему предсказуемее.

Краткая summary-версия для `v1`:

### Frontend baseline

- `Vite + React 19 + TypeScript 5.x`
- `React Router`
- `shadcn/ui + Tailwind CSS 4`
- `Zustand + TanStack Query`
- `TanStack Table + TanStack Virtual`
- `Lightweight Charts`
- `Recharts`

### Backend / Control API baseline

- `Python 3.12+`
- `FastAPI`
- `Pydantic v2 + pydantic-settings`
- `SQLAlchemy 2.x + Alembic`
- `httpx`
- `APScheduler`

### Storage baseline

- `PostgreSQL 16+`
- `TimescaleDB 2.x`
- `pgvector` не является day-one зависимостью и нужен как обязательный слой только при включении `knowledge/retrieval`

### Model training and inference

- `Google Colab` на старте
- позже при необходимости `Lightning AI`
- локальный inference только для compact forecast models
- cloud provider layer должен оставаться сменным и не привязанным к одному вендору

### Runtime and transport principles

- базовые сервисы живут фоном;
- тяжёлые аналитические процессы запускаются по режиму;
- `HTTP/JSON` использовать для control/config/history операций;
- `SSE` считать стандартным transport для server-to-client streams;
- `WebSocket` использовать только там, где действительно нужна двусторонняя realtime-коммуникация.

## 22. Требования к железу и ограничения v1

Исходное железо пользователя:

- `GTX 1660 SUPER 6GB`
- `32 GB RAM`
- `Intel i5-4590`

Инженерное решение:

- локально не пытаться крутить тяжелые LLM;
- локально держать только компактный forecast inference и системные сервисы;
- heavy training держать вне локальной машины;
- cloud text-models использовать как внешний интеллект.

## 23. Нефункциональные требования

- прозрачность и объяснимость
- быстрый переход между режимами
- устойчивость к сбоям облака
- воспроизводимость действий
- наблюдаемость состояния системы
- локальная независимость UI и истории
- контролируемое потребление ресурсов
- отсутствие скрытой автоматики в критичных действиях

## 26. Итог

`CLAY Mission Control v1` — это не бот и не просто dashboard. Это локальный центр управления торгово-аналитической системой, где:

- ИИ-модули работают как специализованные помощники;
- главный агент синтезирует решение;
- пользователь сохраняет контроль;
- панель управляет процессами, режимами, сигналами, историей и анализом;
- система может безопасно переживать деградации;
- demo-обкатка встроена в архитектуру как обязательный этап перед live-режимом.
