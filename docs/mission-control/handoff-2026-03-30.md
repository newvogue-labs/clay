## 2. Зафиксированные архитектурные решения

### 2.1 Общая позиция

- Система должна иметь независимую архитектуру.
- `OpenClaw` не должен быть core foundation для trading system.
- Критичные части системы должны жить вне нестабильной агентной платформы.

### 2.2 Роль OpenClaw

- `OpenClaw` допускается только как опциональный `sidecar/control-plane`.
- Его возможные роли:
  - orchestration дополнительных агентов;
  - memory/RAG слой;
  - operator UI / Mission Control integration;
  - research helper;
  - журналирование reasoning и approvals.
- `OpenClaw` не должен быть единственным runtime для:
  - market data ingestion;
  - risk engine;
  - signal engine;
  - execution path;
  - критичных decision loops.

### 2.3 Базовый подход к alpha

- Зафиксирован `hybrid alpha`.
- Это означает сочетание:
  - market data;
  - technical/statistical features;
  - news/sentiment/event data;
  - LLM-аналитики как верхнего reasoning-слоя.
- Книги, курсы, документация и иные обучающие материалы должны использоваться как `research/RAG/policy layer`, а не как прямой источник торгового сигнала.

### 2.4 Принцип human-in-the-loop

- На текущем видении пользователь самостоятельно совершает сделки.
- Система должна генерировать прогнозы, сигналы, объяснения и рекомендации.
- Будущий execution layer не должен быть обязателен для v1 planning scope.
